## ✅ **React Patterns That Work**

### **React Context for Global State Management**
**When to Use**: Instead of DOM manipulation or custom event systems
**Success Story**: Hotkey toggle feature (PR #5219)
**Pattern**:
```typescript
// Create Context with proper TypeScript typing
interface ContextType {
  state: boolean;
  setState: (value: boolean) => void;
}

const Context = createContext<ContextType | undefined>(undefined);

// Provider component with proper React hooks
export function Provider({ children }: { children: React.ReactNode }) {
  const [state, setState] = useState(false);

  return (
    <Context.Provider value={{ state, setState }}>
      {children}
    </Context.Provider>
  );
}

// Custom hook for consumer components
export function useHook() {
  const context = useContext(Context);
  if (!context) throw new Error('Must use within Provider');
  return context;
}
```

**Why It Works**:
- Eliminates need for DOM manipulation (`element.click()`)
- Avoids custom event system complexity
- Provides proper TypeScript safety
- React's built-in state management handles all edge cases
- Components communicate cleanly through props/context

## ✅ **Hotkey Implementation Patterns**

### **User-Configurable Hotkeys**
**When to Use**: When keyboard shortcuts need to be user-customizable
**Success Story**: Tool output expansion hotkey (PR #5219)
**Pattern**:
```typescript
interface HotkeyConfig {
  key: string;
  ctrl: boolean;
  meta: boolean;
  shift: boolean;
  alt: boolean;
}

// Storage in localStorage for persistence
const saveHotkey = (hotkey: HotkeyConfig) => {
  localStorage.setItem('hotkey', JSON.stringify(hotkey));
};

// Proper hotkey matching
const matchesHotkey = (event: KeyboardEvent, hotkey: HotkeyConfig): boolean => {
  return (
    event.key.toLowerCase() === hotkey.key.toLowerCase() &&
    event.ctrlKey === hotkey.ctrl &&
    event.metaKey === hotkey.meta &&
    event.shiftKey === hotkey.shift &&
    event.altKey === hotkey.alt
  );
};
```

**Why It Works**:
- Users can customize without code changes
- Preferences persist across sessions
- No interference with browser defaults
- Proper cross-platform support (Ctrl vs Cmd)

## ❌ **Anti-Patterns to Avoid**

### **DOM Manipulation in React**
**Problem**: Using `element.click()` or direct DOM manipulation
**Example of What NOT to Do**:
```typescript
// ❌ ANTI-PATTERN
const collapsedButtons = document.querySelectorAll('button[aria-label="Expand"]');
collapsedButtons.forEach(element => {
  if (element instanceof HTMLElement) {
    element.click(); // This breaks React's virtual DOM
  }
});
```

**Better Approach**: Use React state and props to control UI state

### **Custom Event Systems**
**Problem**: Creating custom events when React Context suffices
**Example of What NOT to Do**:
```typescript
// ❌ ANTI-PATTERN
const toggleEvent = new CustomEvent('toggleAll', { detail: { expand: true } });
document.dispatchEvent(toggleEvent);

// Then listening elsewhere
document.addEventListener('toggleAll', handleToggle);
```

**Better Approach**: Use React Context for component communication

### **Browser-Conflicting Hotkeys**
**Problem**: Using hotkeys that interfere with browser behavior
**Example**: Ctrl+R (refresh), Ctrl+S (save), Ctrl+F (find)
**Better Approach**: Use less common combinations like Ctrl+E, Ctrl+T, or make configurable

## ✅ **Testing and Validation**

### **TypeScript Compilation as Validation**
**Pattern**: Use `bun build --no-bundle` for rapid type checking
**Command**:
```bash
bun build ui/desktop/src/components/ComponentName.tsx --no-bundle
```

**Why It Works**:
- Catches type errors before runtime
- Faster than full build process
- Validates imports and component structure
- Works on individual files or full app

## 📋 **Project Architecture Insights**

### **Component Structure**
- `ui/desktop/src/components/` - Reusable UI components
- `ui/desktop/src/hooks/` - Custom React hooks
- `ui/desktop/src/contexts/` - React contexts for state management
- `ui/desktop/src/components/settings/` - Settings-related components
- `ui/desktop/src/components/ui/` - Low-level UI primitives

### **State Management Priorities**
1. **Local State**: useState for component-internal state
2. **Context API**: For global or cross-component state
3. **Avoid Redux**: Unless absolutely necessary for complex state

### **Styling Approach**
- **Tailwind CSS**: Primary styling solution
- **CSS-in-JS**: For dynamic theming
- **Styled Components**: Where complex styling needed
